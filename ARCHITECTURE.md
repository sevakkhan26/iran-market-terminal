# Iran Market Terminal — Architecture Review & v2 Design

This document reviews the v1 implementation, explains every v2 design decision, and lays out the roadmap toward a commercial-grade product. It covers the seven areas requested: product architecture, missing features, UI structure, database, backend, frontend, and professional trading features.

## 1. Improved product architecture

V1 was a competent MVP with a fundamental limitation: it only ever saw the top of the order book. Every exchange connector fetched a single best bid and best ask, which means liquidity, depth, volume, slippage, and realistic arbitrage sizing were structurally impossible — the "Liquidity Discovery Engine" branding had no liquidity data behind it. V2 restructures the data pipeline around full-depth order books (top 20 levels), 24-hour ticker statistics, and a global USD reference price, flowing through four layers: connectors (async fetch, normalization to Toman), an aggregator (composite index, snapshot persistence), a metrics engine (in-memory ring buffers for fast rolling statistics), and an alert engine (rule evaluation with cooldowns). A WebSocket fan-out pushes live state and alert events to the UI; REST serves everything else. The whole backend remains a single FastAPI process — deliberately, since six venues at a 3-second cadence is far below the threshold where message queues or worker processes pay for themselves. The escape hatches are documented in the roadmap below.

Two v1 problems this dissolves: the dead code (candle connectors, per-exchange history, calendar table — all written but never wired) is now fully integrated, and the two parallel frontends (Streamlit + vanilla JS) are replaced by one React application served by the backend.

## 2. Missing features — now built

The most important gap was economic: v1 had no USDT/IRT market and no international reference price, so it could not answer the single most-watched number on an Iranian desk — the Iran premium. V2 adds USDT as a first-class asset, pulls global USD prices from CoinGecko (keyless, configurable), and computes the implied-USD premium per asset: `(asset_TMN / USDT_TMN) / asset_USD − 1`. Premium history is persisted and charted.

Arbitrage in v1 was a client-side O(n²) gross-price comparison. V2 computes it server-side, walks both books level-by-level to find the depth-limited executable size, and applies per-exchange taker fees from config — reporting gross %, net %, max size, and estimated profit. A 0.4% gross edge that dies under two 0.25% taker fees now correctly shows as no opportunity.

Also added: liquidity scores (0–100, relative: 50% log-scaled depth, 35% spread tightness, 15% freshness), depth imbalance, 1H/24H/7D changes (ring buffers for short windows, SQLite for 7D), spread statistics (mean/min/max/σ over any window), anomaly detection (cross-venue price deviation z-scores, stale feeds, liquidity drops vs 1-hour average), a persistent alert engine with eight rule types, real OHLC candles (native klines where exchanges expose them, ring-built 60-second candles everywhere else including the composite index), calendar surprise % with per-indicator historical surprise statistics, and a demo mode that generates realistic synthetic markets so the product can be developed and demonstrated with no connectivity.

## 3. Better UI structure

The UI is reorganized around desk workflows rather than data sources. Markets (CMC-style ranked table: price, 1H/24H/7D pills, premium, min spread, liquidity bar, volume, best-buy/best-sell venue, 7-day sparkline) answers "what is happening." Clicking a row opens the Market Detail view — candlestick chart with 1H/4H/1D/1W/1M ranges and per-venue/composite source switching, the full exchange comparison table (bid/ask, spread + 1h stats, depth, imbalance, liquidity score, volume, fees, latency, status), a cumulative depth chart, and the pair's fee-adjusted arbitrage ladder — answering "where is the best price." The Dealing Desk page is the risk view: arbitrage scanner, largest movers, widest spreads, liquidity warnings, live alert feed, and the rule manager. Analytics goes deep on one asset (spread history multi-line chart per venue, liquidity distribution, premium history, anomaly log). Calendar and News carry market intelligence, and Admin holds data management.

A persistent stats bar (BTC price, USDT/TMN, global BTC, premium, total volume, live venue count) keeps global context visible on every page — the Bloomberg pattern. The design language is a dark navy terminal theme with tabular numerals, red/green semantics, status dots, and skeleton loading states. The UI is bilingual (EN/FA) with full RTL mirroring via logical CSS properties, and Persian dates render in the Jalali calendar automatically through `Intl` — no conversion library.

## 4. Database improvements

V1 opened a fresh SQLite connection per insert, wrote ~518k rows/day at the 2-second cadence, and never deleted anything; the calendar table existed but was never written, and per-exchange history was written but never read. V2 keeps one long-lived WAL connection behind a lock with batched transactional writes. Hot data (every cycle) lives in in-memory ring buffers; the database receives 5-minute snapshots (per-exchange and composite) — a ~150× write reduction that still supports every historical metric. New tables: `candles` (upserted OHLC, multi-resolution), `calendar_events` (actually written now, with surprise %, unique-keyed so re-fetches update actuals in place — this is what enables historical surprise stats), `alert_rules` / `alert_events`, and `custom_exchanges` / `custom_pairs` for runtime-added sources. An hourly retention job prunes every table to configured horizons (90d snapshots, 365d candles, 30d alerts by default). The data directory is relocatable via `TERMINAL_DATA_DIR`.

The natural next step at higher scale is TimescaleDB or ClickHouse for tick storage, but SQLite-WAL with this write pattern comfortably handles this workload for years of data.

## 5. Backend improvements

Beyond the pipeline redesign: proper `logging` throughout instead of `print`; per-connector timeouts with degenerate-book validation (a crossed or empty book is rejected rather than displayed); ticker stats fetched on a slower cadence (60s) than books; the reference price cached for 120s; snapshot writes offloaded to threads; alert evaluation moved off the hot path (every ~5 cycles, in a worker thread, with thread-safe WebSocket marshalling via `call_soon_threadsafe`). Settings changes persist to disk and apply without restart. CORS is no longer `*`-with-credentials — it is scoped to the dev origin, and mutating endpoints (settings, alert rules, admin CRUD) accept an optional `X-Admin-Token` guard. The v1 `min_impact` bug (parameter accepted, never applied) is fixed, as is the silent asset coercion in the history endpoint. Unit tests cover the financial math — premium, fee/depth-aware arbitrage, liquidity scoring, change computation, numeric parsing of calendar values ("250K", "−3.8%"), and candle resampling.

The declarative `GenericRestConnector` is the data-management answer: a new exchange is a JSON spec (order-book URL, dot-paths to bids/asks, symbol template, optional Rial→Toman scale, optional stats endpoint) submitted through the Admin UI, stored in SQLite, and hot-loaded into the connector pool — no code, no rebuild, no restart. Pairs work the same way.

## 6. Frontend improvements

V1's vanilla JS avoided CDNs because Cloudflare-hosted libraries are often unreachable from Iran — a constraint worth honoring, not discarding. V2 keeps the guarantee while upgrading the stack: React 18 + Vite with every dependency (React, TradingView's lightweight-charts, Inter and Vazirmatn fonts) bundled into the build output. Zero runtime CDN requests; the built app works fully offline against the local backend. Components are small and composable (ChangePill, Sparkline, ScoreBar, Seg, Chips, Countdown, sortable tables). Charts are professional-grade: candlesticks with volume histograms, multi-series spread lines per venue, area charts for premium, and a custom canvas depth chart. Live updates arrive over WebSocket with automatic reconnect (alert toasts appear globally); page data polls REST at per-page cadences. Language and layout direction switch instantly and persist in `localStorage`.

## 7. Professional trading features to add next

What a desk would ask for next, roughly in order of value: **execution-aware analytics** — slippage curves per venue (cost of executing 100M/500M/1B TMN market orders from stored depth), which turns the liquidity score into a tradable number; **transfer-aware arbitrage** — withdrawal fees, network transfer times, and deposit status per venue, since spot arb across Iranian exchanges is settlement-constrained; **VWAP/TWAP references and realized volatility** per venue for execution benchmarking; **order-book replay** from stored snapshots for post-mortem analysis; **alert delivery channels** (Telegram is the natural one for Iranian desks); **USD/IRT unofficial rate ingestion** to decompose the premium into FX component vs crypto component; **watchlists and multi-desk auth** (JWT, roles) once more than one trader uses it; **Prometheus metrics + healthchecks** for operating it as a service; and **maker/taker fee tiers and inventory tracking** if the desk starts executing against the signals. The architecture supports all of these without structural change: they are new services on the existing pipeline and new cards on the existing pages.
